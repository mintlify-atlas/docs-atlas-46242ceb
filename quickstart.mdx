---
title: "Quickstart"
description: "Install the server, configure environment variables, and run your first visual parts search."
---

<Steps>
  <Step title="Install dependencies">
    Clone the repository and install the Python dependencies. A virtual environment is recommended.

    ```bash
    python -m venv .venv
    source .venv/bin/activate   # Windows: .venv\Scripts\activate
    pip install -r requirements.txt
    ```

    The key packages installed are:

    | Package | Purpose |
    |---|---|
    | `fastapi`, `uvicorn[standard]` | HTTP server |
    | `torch`, `torchvision` | DINOv2 inference |
    | `sqlalchemy`, `psycopg2-binary`, `pgvector` | Database ORM and vector search |
    | `cloudinary` | Image storage for parts |
    | `pillow`, `pillow-heif` | Image decoding (JPEG, PNG, WebP, HEIC) |
    | `slowapi` | Rate limiting |
  </Step>

  <Step title="Configure environment variables">
    Create a `.env` file in the project root with the following variables. The server will not start if the database URLs are missing.

    ```bash .env
    # PostgreSQL connection strings (must have the pgvector extension enabled)
    DATABASE_URL_PLANTA1=postgresql://user:password@localhost:5432/planta1_db
    DATABASE_URL_PLANTA2=postgresql://user:password@localhost:5432/planta2_db

    # Cloudinary credentials (used to store part images)
    CLOUDINARY_CLOUD_NAME=your_cloud_name
    CLOUDINARY_API_KEY=your_api_key
    CLOUDINARY_API_SECRET=your_api_secret
    ```

    <Info>
    Each database must have the `pgvector` extension installed. Run `CREATE EXTENSION IF NOT EXISTS vector;` on each database before starting the server.
    </Info>
  </Step>

  <Step title="Start the server">
    <CodeGroup>

    ```bash Development
    uvicorn app:app --reload --host 0.0.0.0 --port 8000
    ```

    ```bash Production
    uvicorn app:app --host 0.0.0.0 --port 8000 --workers 2
    ```

    </CodeGroup>

    The API will be available at `http://localhost:8000`. Interactive documentation is served at `http://localhost:8000/docs`.

    <Note>
    On first startup the DINOv2 ViT-G/14 model is downloaded from `torch.hub` (approximately 3.5 GB) and a warmup inference is run. This can take several minutes the first time.
    </Note>
  </Step>

  <Step title="Create a machine">
    Before adding parts you must create at least one machine. All parts belong to a machine.

    ```bash
    curl -X POST http://localhost:8000/maquinas/ \
      -H "x-planta: planta 1" \
      -F "clave=CNC-001" \
      -F "nombre=Torno CNC Mazak" \
      -F "descripcion=Torno de control numérico línea A" \
      -F "ubicacion=Nave 3, Sector B" \
      -F "uso_en=Manufactura"
    ```

    Successful response:

    ```json
    {
      "ok": true,
      "data": {
        "clave": "CNC-001",
        "nombre": "Torno CNC Mazak",
        "descripcion": "Torno de control numérico línea A",
        "ubicacion": "Nave 3, Sector B",
        "uso_en": "Manufactura",
        "proveedores": null,
        "tiene_foto": false,
        "imagen": null
      }
    }
    ```
  </Step>

  <Step title="Add a part with an image">
    Add a part by posting its metadata and at least one image. The server extracts a 1536-dimensional embedding from the image and stores both the image (via Cloudinary) and the embedding (via pgvector).

    ```bash
    curl -X POST http://localhost:8000/piezas/agregar \
      -H "x-planta: planta 1" \
      -F "clave=PIEZA-BUJE-42" \
      -F "nombre=Buje de bronce 42mm" \
      -F "maquina_id=CNC-001" \
      -F "ubicacion=Almacén A-04" \
      -F "uso_en=Eje principal" \
      -F "proveedores=Metalúrgica Norte" \
      -F "imagen=@/path/to/buje.jpg"
    ```

    Successful response:

    ```json
    {
      "ok": true,
      "clave": "PIEZA-BUJE-42",
      "msg": "Pieza guardada exitosamente"
    }
    ```

    <Tip>
    You can supply up to three images per part (`imagen`, `imagen_2`, `imagen_3`). Each gets its own embedding. During search, the system uses `func.least()` across all three embeddings, so parts with multiple angles match more reliably.
    </Tip>
  </Step>

  <Step title="Run a visual search">
    Upload a query photo to find the most visually similar parts across the entire plant catalog.

    ```bash
    curl -X POST http://localhost:8000/piezas/buscar/global \
      -H "x-planta: planta 1" \
      -F "imagen=@/path/to/query.jpg" \
      -F "limite=5"
    ```

    Response:

    ```json
    {
      "ok": true,
      "data": [
        {
          "clave": "PIEZA-BUJE-42",
          "nombre": "Buje de bronce 42mm",
          "maquina": "Torno CNC Mazak",
          "imagen": "https://res.cloudinary.com/your_cloud/image/upload/piezas/PIEZA-BUJE-42_imagen_...",
          "distancia": 0.1823,
          "nivel": "Alta",
          "detalles": {
            "ubicacion": "Almacén A-04",
            "uso_en": "Eje principal",
            "proveedores": "Metalúrgica Norte",
            "tiene_foto": true,
            "imagenes_extra": [null, null]
          }
        }
      ]
    }
    ```

    The `nivel` field indicates match confidence based on L2 distance:

    | `nivel` | Distance range | Meaning |
    |---|---|---|
    | `Alta` | < 0.35 | High confidence match |
    | `Media` | 0.35 – 0.55 | Possible match, verify visually |
    | `Baja` | ≥ 0.55 | Low confidence |
  </Step>

  <Step title="Search with ROI (optional)">
    If only part of the image contains the part you want to match, pass ROI (Region of Interest) coordinates as fractional values (0.0–1.0) relative to image dimensions.

    ```bash
    curl -X POST http://localhost:8000/piezas/buscar/global \
      -H "x-planta: planta 1" \
      -F "imagen=@/path/to/query.jpg" \
      -F "limite=5" \
      -F "x=0.25" \
      -F "y=0.10" \
      -F "w=0.50" \
      -F "h=0.60"
    ```

    When ROI coordinates are provided, the server computes a hybrid embedding: **60% ROI crop + 40% full image**, then re-normalizes. This improves precision when the target part occupies a small area of the frame.
  </Step>
</Steps>
